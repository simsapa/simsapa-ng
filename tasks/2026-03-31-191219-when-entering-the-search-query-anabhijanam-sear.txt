  Here's the summary of the fix:

  Root cause: The pli_stem tokenizer applies the PńĀli Snowball stemmer before
  ASCII folding. The stemmer's suffix rules use diacritical characters (e.g.,
  ńĀnaŠĻĀ, ŇęnaŠĻĀ), so when an ASCII query like anabhijanam passes through, the
  stemmer doesn't recognize the ASCII suffix anam and produces a different token
   than what was indexed from anabhijńĀnaŠĻĀ. The search required the stemmed
  content field to match (Occur::Must), so no results were returned.

  Fix (backend/src/search/searcher.rs:351-365): Changed the boolean query logic
  from:
  - content = Must, content_exact = Should (boost only)

  To:
  - A nested BooleanQuery where both content and content_exact are Should
  clauses (tantivy requires at least one Should to match), wrapped in a Must in
  the outer query

  This way, an ASCII query can match via the content_exact field (which uses the
   {lang}_normalize tokenizer with ASCII folding but no stemming), even when the
   stemmed content field doesn't match. When both match (e.g., with diacritical
  queries), the boosted content_exact still improves ranking.
