  What's actually slow

  Path A (DPD) is already FTS5-backed for the substring scan against
  dpd_headwords_fts.lemma_1 ‚ÄĒ the slowness isn't there. The bottleneck is the
  N+1 pattern that follows: for each matched DPD headword (line 1863‚Äď1894), it
  runs a separate dict_words Diesel query to resolve word == lemma_1. For a
  query like "gacchati" that matches hundreds of inflected DPD lemmas, that's
  hundreds of sequential SQLite round-trips across two database connections.

  Path B (user dicts) is already a single FTS5 query and is fine.

  Key observation

  For dict_label = "dpd", the value of dpd_headwords.lemma_1 is exactly the same
   as dict_words.word. The dpd_headwords_fts index is redundant for headword 
  match ‚ÄĒ dict_words_fts.word already covers every DPD lemma plus every
  user-dict headword. The whole two-path design can collapse into one FTS query
  against dict_words_fts.word.

  Recommended plan

  1. Collapse Path A into Path B. Replace the entire function body with one FTS
  query: dict_words JOIN dict_words_fts f ON f.rowid = dw.id WHERE f.word LIKE 
  ?, gated by the existing dict_source_uids filter. This eliminates the N+1 and
  the cross-DB hop ‚ÄĒ partial matches like saŠĻÖkh ‚Üí saŠĻÖkhńĀra, abhisaŠĻÖkhńĀra already
   work because trigram-tokenized LIKE '%saŠĻÖkh%' is index-served by FTS5.
  2. Add ASCII folding. Add a word_ascii column to dict_words_fts in
  scripts/dictionaries-fts5-indexes.sql (and to the bootstrap insert + the
  INSERT/UPDATE triggers). Populate it via a Rust-side ASCII-fold pass during
  bootstrap (there's already an ascii_fold style helper used for
  dpd_headwords.word_ascii ‚ÄĒ reuse it). The query becomes WHERE f.word LIKE ?A 
  OR f.word_ascii LIKE ?B where ?B is the ASCII-folded query. Requires
  re-bootstrap of dictionaries.sqlite3. The dpd_headwords_fts script can be
  updated the same way (or dropped entirely if Path A is collapsed).
  3. Re-bootstrap. Per CLAUDE.md, FTS5 schema changes need a manual
  re-bootstrap; the SQL script in scripts/ already DROPs + recreates the FTS
  table and triggers.
